Session Handoff

A fresh Claude Code session starts with an empty context window. These two manual commands carry one task across that gap — /session-handoff saves where you are, /session-resume picks it back up.

Session · days later
A fresh, empty window — refilled from the file in seconds, same topic.

The file is the only thing that crosses the gap. Pick the topic up again and it chains — _01 → _02 → _03, each new file carrying the last one forward so the newest is always the full picture.

Where they sit in the workflow

They don't replace planning or finishing — they live in the pause / resume middle of a single task.

/new-cycle

/end-cycle

How each command works

/session-handoff [topic] [--done]

  1. Gather this session: date, branch, git status, background process IDs, open TODOs, files you touched — plus any superpowers spec/plan paths.
  2. Pick topic + sequence — propose existing topics + a branch suggestion, confirm (no accidental chain-fork).
  3. Carry-forward from the previous handoff so the newest file is the complete current state.
  4. Warn on any secret pattern (reference creds by path, never values).
  5. Ensure .claude/session-handoffs/ is in .gitignore (once).
  6. Write {topic}_{NN}.md.
  7. Closing reflection (propose-only): surface durable facts as Claude-memory candidates, any plan-drift update, and a refresh of any project doc the session left stale — show, ask, write only on confirm.
  8. --done → archive the whole chain to done/.
  9. Confirm + hard stop.

/session-resume [topic] [--all]

  1. Scan the store, group by topic, newest first (skip done/ unless --all).
  2. Pick a topic (or pass one to load it directly).
  3. Staleness heads-up — > 7 days old, branch changed, or working tree moved.
  4. Load the latest file, follow its [READ-AT-RESUME] / plan / spec links, and summarize: where it left off + key files + running state + the full decisions and rejected options.
  5. Propose the single next action — but do not auto-implement.

Step 7 shows you the exact content and waits for your OK — a memory fact, a plan update, a doc refresh. It links by path, never copies a spec, never auto-writes. Resume re-reads the handoff and follows its [READ-AT-RESUME] links to recover the full decisions and roadmap; memory loads on its own each session.

The store & chain-linking

Per project, gitignored. The same topic chains by sequence number, so multi-day work stays in one thread.

<your project>/.claude/session-handoffs/        # gitignored
├── checkout-bug_01.md
├── checkout-bug_02.md      ← same topic, next session (carry-forward)
├── auth-refactor_01.md
└── done/                   # archived via --done, hidden from resume
Why not just /park? buzzwoo's /park is a lighter note-dump, and /resume-bw is plan-centric — it can't cleanly let you pick among several parallel topics. This pair adds a richer schema (running state, verification commands), topic-aware selection, and sequence chain-linking for the "continue the same thing days later" case. It is intentionally decoupled from buzzwoo so it can be tested independently and proposed for adoption later.

Where the design comes from

No single existing tool did the whole job, so this pair borrows the strongest idea from each — and deliberately drops the rest. What each contributed, and where it actually shows up in the commands:
