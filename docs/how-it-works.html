Session Handoff

Two personal Claude Code commands that carry one task across the gap between sessions — /session-handoff saves where you are, /session-resume picks it back up.

New to Claude Code? Every time you open a new session, Claude starts with a blank memory — it can't see what you did yesterday, or even in the session you just closed. Come back to the same task hours or days later and you'd normally re-explain everything from scratch. These two commands write that state to a file so you don't have to.

The file is the only thing that crosses the gap. Pick the topic up again and it chains — _01 → _02 → _03, each new file carrying the last one forward so the newest is always the full picture.

They don't replace planning or finishing — they live in the pause / resume middle of a single task.

How each command works

/session-handoff [topic] [--done]

  1. Gather this session: date, branch, git status, background process IDs, open TODOs, files you touched.
  2. Pick topic + sequence — propose existing topics + a branch suggestion, confirm (no accidental chain-fork).
  3. Carry-forward from the previous handoff so the newest file is the complete current state.
  4. Warn on any secret pattern (reference creds by path, never values).
  5. Ensure .claude/session-handoffs/ is in .gitignore (once).
  6. Write {topic}_{NN}.md.
  7. Closing reflection (propose-only): surface durable facts worth saving to memory, plan drift, or a stale doc — show, ask, write only on confirm.
  8. Confirm + hard stop.

/session-resume [topic] [--all]

  1. Scan the store, group by topic, newest first (skip done/ unless --all).
  2. Pick a topic (or pass one to load it directly).
  3. Staleness heads-up — > 7 days old, git branch changed, or the working tree moved since the save (current git status compared against the snapshot stored in the handoff).
  4. Load the latest file, follow its plan / spec links, and summarize: where it left off + key files + running state + the full decisions and rejected options.
  5. Propose the single next action — but do not auto-implement.

It shows you the exact content and waits for your OK — a memory fact, a plan update, a doc refresh. It links by path, never copies, never auto-writes.

The store & chain-linking

Per project, gitignored. The same topic chains by sequence number, so multi-day work stays in one thread.

<your project>/.claude/session-handoffs/        # gitignored
├── checkout-bug_01.md
├── checkout-bug_02.md      ← same topic, next session (carry-forward)
├── auth-refactor_01.md
└── done/                   # archived via --done, hidden from resume
Why /park wasn't enough for me. I already use buzzwoo's /park — but for my own way of working it stopped short in three places. It's a light note-dump, so when I came back days later I still had to reconstruct the running state (which server, which verification command, which files). /resume-bw is plan-centric, so it couldn't cleanly let me pick between several parallel topics I had going at once. And nothing chained one topic across many sessions. So I built this pair for that specific gap — a richer schema, topic-aware selection, and sequence chain-linking — and kept it decoupled from buzzwoo so I could test it on its own. It sits next to /park, it doesn't replace it.

A quick look at what these actually are — no deep dive needed to use them.

Just two Markdown files

Each command is a single Markdown file: a small frontmatter header telling Claude Code how to run it, followed by numbered instructions in plain English. No code, no build step.

Frontmatter — the header

  • description — what the command does + the rule that it only runs when you ask.
  • argument-hint — the optional arguments, e.g. [topic-slug] [--done].
  • allowed-tools — the only tools it may use (read/write files, ask you a question, run a shell command). Nothing else is permitted.

Source here, live over there

Claude Code loads personal commands from your home folder, not from this repo. So the repo holds the source, and installing means copying the two files to the live location.

Rule: edit the source, then deploy. A check makes sure the live copy hasn't drifted from the source — source == deployed.

Safe on Windows

Files are written with the editor's file tools rather than shell tricks, and the read-only checks batch through PowerShell where a plain chained shell command would trip up. Nothing about the commands assumes macOS or Linux.

Git: read and compare, never commit

Saving records the current git branch and a git status snapshot in the handoff header; resuming runs the same read-only checks again and compares — that's where the staleness note comes from. Neither command ever commits or pushes (that stays with /end-cycle); the only git writes are the one-time .gitignore entry and the git mv that archives a finished topic.

Tested, not just written

Both commands ship with two layers of tests so changes don't quietly break them:

  • Static checks — a script validates each file's structure, its header fields, the step numbering, and that the deployed copy matches the source.
  • Behavioural checks — Claude runs the commands for real in a throwaway sandbox (save → carry a topic forward → resume), then a script inspects the files they produced to confirm they behaved: propose-only, read-only on resume, correct chaining.

The full decision log and test details live in the repo (plan/ and tests/) — this page is just the overview.
